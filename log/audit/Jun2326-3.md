## 1. 🔴 Security Issues

### 1.1 CRITICAL — Hardcoded API Key in a Public Repository

**File:** `JavaScript/maintenance.js` line 209

```javascript
let apiKey = "AIzaSyAVnHFCrWdeB1SsUymal-nvENSqvYnPMWw";
```

A Google Gemini API Key is hardcoded directly in a **public repository**. Anyone can see and use this key, opening the project to quota theft or unexpected billing charges.

**Fix:**
- Remove the key from source code immediately and revoke it in Google Cloud Console.
- Move the key to the backend (Google Apps Script) and call the API server-side.
- If client-side usage is required, proxy all Gemini requests through a GAS endpoint instead.

---

### 1.2 HIGH — No Subresource Integrity (SRI) on CDN Links

Every page loads external scripts from CDNs without an `integrity` attribute:

```html
<!-- Insecure -->
<link href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.2/dist/css/bootstrap.min.css" rel="stylesheet">
<script src="https://cdn.auth0.com/js/auth0-spa-js/2.0/auth0-spa-js.production.js"></script>
```

If a CDN is compromised or the URL is hijacked, malicious code runs directly in the user's browser.

**Fix:**
```html
<link href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.2/dist/css/bootstrap.min.css"
      rel="stylesheet"
      integrity="sha384-T3c6CoIi6uLrA9TneNEoa7RxnatzjcDSCmG1MXxSR1GAsXEV/Dwwykc2MPK8M2HN"
      crossorigin="anonymous">
```

---

### 1.3 HIGH — No Content Security Policy (CSP)

There is no CSP HTTP header or `<meta>` tag on any page, leaving the site exposed to XSS attacks.

**Fix:** Add a meta tag inside every `<head>`:
```html
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self' https://cdn.auth0.com https://cdn.jsdelivr.net https://apis.google.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com https://cdn.jsdelivr.net https://cdnjs.cloudflare.com; img-src 'self' data: https:; connect-src 'self' https://*.auth0.com https://script.google.com;">
```

---

### 1.4 MEDIUM — Auth0 Credentials Exposed in Public Code

**File:** `JavaScript/auth.js`

```javascript
const AUTH0_DOMAIN = "litalkeducation.us.auth0.com";
const AUTH0_CLIENT_ID = "PGqozL94LzOwstm4pD39W5kvalYRiK7w";
```

While SPA Client IDs are designed to be public, exposing the domain alongside the Client ID makes it easier for attackers to craft convincing phishing pages.

**Fix:**
- Lock down Allowed Callback URLs and Allowed Web Origins to exact values in the Auth0 Dashboard.
- Enable Attack Protection in Auth0.

---

### 1.5 MEDIUM — Access Token Sent in Request Body

**File:** `JavaScript/auth.js` — `saveProfileData()` function

```javascript
const payload = {
    access_token: token,  // ⚠️ token in request body
    name: nameVal,
    ...
};
```

Best practice is to pass the token via an `Authorization: Bearer` header to prevent it from appearing in server logs.

### 2.3 Service Worker: Cache-First with No Good Fallback

**File:** `service-worker.js`

The current strategy is cache-first only. If cached files are stale, users won't see updated content until they manually clear their cache.

```javascript
// Current: cache hit → return immediately, no freshness check
if (response) {
    return response;
}
```

**Fix:** Use a Stale-While-Revalidate strategy for HTML pages.

---

### 2.4 Google Fonts Missing `font-display: swap`
